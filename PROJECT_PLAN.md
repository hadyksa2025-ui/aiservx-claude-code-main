# PROJECT_PLAN.md

> Persistent, living plan for transforming `open-claude-code-main` into a small
> AI IDE platform ("Open Claude Code Desktop") powered by a hybrid
> **OpenRouter (planner) + Ollama (executor)** loop. Updated by the agent as
> work progresses. Treat this file as the **source of truth** for architecture,
> progress, and next steps — together with `PROJECT_MEMORY.json`.

---

## 0. Provenance & Scope Constraints

  We must not modify, depend on, or import anything from `src/`.
- All product work lives under `desktop/`. The Tauri project root is
  `desktop/src-tauri/`. The frontend is served from `desktop/dist/`
  (generated by Vite from `desktop/frontend/`).
- The old `desktop/src/` (duplicate Rust files with a `run_claude` command
  that shells out to `src/entrypoints/cli.tsx`) is **obsolete** and not part
  of the Tauri build. We treat it as dead code and do not extend it.

## 1. Success Criteria

The finished system must be able to:

1. Open a local project directory.
2. Show the project's file tree and auto-refresh it on disk changes.
3. Accept user chat prompts.
4. Decide what to do with a hybrid planner/executor model pair.
5. Execute tool calls (read/write files, list directory, run shell command)
   in the actual file system of the opened project.
6. Stream tool calls, file diffs and model tokens to the UI in real time.
7. Persist project memory to `PROJECT_MEMORY.json` so state survives restarts.
8. Gracefully fall back to Ollama-only if OpenRouter is unavailable.

```
├───────────────────────────────────────────────────────────────┤
│                     Backend Layer (Tauri/Rust)                │
│   fs::list_dir  fs::read_file  fs::write_file  fs::watch_dir  │

├───────────────────────────────────────────────────────────────┤
│                         AI Layer                              │

├───────────────────────────────────────────────────────────────┤
│                       Tool Runtime                            │
│   read_file  write_file  list_dir  run_cmd                    │
│   (all executed by Rust inside the opened project root)       │
└───────────────────────────────────────────────────────────────┘
```

### 2.1 Data flow

```
user prompt
  → ai::send_chat { project_dir, message, history }
  → Router picks Planner (OpenRouter) or Executor (Ollama)
  → model returns assistant text and/or tool_call[]
  → for each tool_call: Rust executes tool → emits tool:call + tool:result
  → result is appended to history, loop continues
  → when model returns assistant text with no tool_call, emit ai:done
```

### 2.2 Channels / events

- `ai:token`            — streaming model tokens `{ text, role }`
- `ai:tool_call`        — `{ id, name, args, role }`
- `ai:tool_result`      — `{ id, ok, output, diff, role }`
- `ai:step`             — `{ index, role, title, status }` timeline updates
- `ai:confirm_request`  — `{ id, cmd, project_dir, timeout_ms }` shell-cmd gate
- `ai:error`            — `{ message, role? }`
- `ai:done`             — `{ assistant, iterations }`
- `fs:changed`          — `{ path, kind }` from the directory watcher

## 3.6 Autonomous Task Engine (PR #3)

- `controller::start_goal(project_dir, goal)` is the new top-level
  command. It (1) scans the project and persists a `project_map`,
  (2) asks the planner for a JSON task list (capped by
  `settings.max_total_tasks`), (3) runs each task through
  `ai::run_chat_turn` with a goal-aware wrapper prompt, (4) consults the
  reviewer for an `OK:` / `NEEDS_FIX:` verdict, (5) retries up to
  `settings.max_retries_per_task` times with the reviewer's instruction
  fed back as feedback, and (6) archives the completed tree into
  `task_history[]`.
- `AppState.goal_cancelled` is a separate flag from `cancelled` so that
  in-flight chat turns can be cancelled without killing the whole goal,
  and a goal can be stopped without surprising the next chat turn.
- Events emitted for the UI:
  - `project:scan_done` — `{ languages, entry_points, configs, deps, file_count }`
  - `task:goal_started` — `{ id, goal, task_count }`
  - `task:added`        — `{ goal_id, task }`
  - `task:update`       — `{ goal_id, id, status, retries?, result? }`
  - `task:goal_done`    — `{ id, goal, status, completed, failed }`
  - `task:failure_logged` — `{ task_id, error }`
- PROJECT_MEMORY.json gets four new top-level slots:
  - `active_task_tree` — the in-flight tree (null once a goal finishes).
  - `task_history[]`   — completed / failed / cancelled trees (cap 200).
  - `failures_log[]`   — `{ at, task_id, error }` entries (cap 200).
  - `project_map`      — latest scanner snapshot.

## 4. Design Decisions

- **Frontend framework**: Vite + React + TypeScript. Rationale: smallest
  working SPA that still gives a proper component model, and Tauri v2 has
  first-class support for Vite out of the box.
- **Styling**: plain CSS, scoped under `desktop/frontend/src/styles.css`. No
  component library to keep the first PR lean.
- **Tauri v2** with `@tauri-apps/api` v2 for `invoke` + `event.listen`.
- **Tool calling**: OpenAI-compatible tool schema, reused by both
  OpenRouter and Ollama (Ollama supports the same schema on modern models
  via `/api/chat` with `tools:[]`). Fallback parser supports models that
  emit ```<tool_call>{...}</tool_call>``` blocks.
- **Router**: if `OPENROUTER_API_KEY` is configured, the first turn of each
  user request goes to the planner (OpenRouter) which produces a short plan
  and optionally tool calls; subsequent turns go to the executor (Ollama).
  If OpenRouter fails at any point, we silently fall back to executor-only.
- **Watcher**: `notify` crate in debounced mode, one watcher per opened
  project directory, events debounced to 150ms before emitting to the UI.
- **Memory**: `PROJECT_MEMORY.json` is written by Rust on every chat turn.
  It is split into `session` (ephemeral, last N messages) and `project`
  (persistent index of files opened and tools used).

## 5. Non-goals (for this PR)

- Full language-server integration / code intelligence.
- Multi-file diff viewer with 3-way merge.
- Authentication flows (OpenRouter key is read from env or saved to disk
  via a simple `save_settings` command).
- Windows installer tuning; we only guarantee `cargo tauri dev` works.

## 6. How to run

```bash
# prerequisites: Node >= 18, Rust stable, Ollama (optional), OpenRouter key (optional)

cd desktop/frontend
npm install
npm run build         # produces desktop/dist/index.html + assets

# then either:
cd ../src-tauri
cargo tauri dev       # dev mode

# or for a headless smoke-check of the Rust side:
cargo check
```

Environment variables:

| Variable            | Purpose                                    | Default                       |
|---------------------|--------------------------------------------|-------------------------------|
| `OPENROUTER_API_KEY`| Enables the planner path                   | unset → executor-only         |
| `OPENROUTER_MODEL`  | Model id for the planner                   | `openrouter/auto`             |
| `OLLAMA_BASE_URL`   | Ollama base url                            | `http://localhost:11434`      |
| `OLLAMA_MODEL`      | Executor model                             | `llama3.1:8b`                 |

Settings can also be set at runtime via the UI and are persisted to the
standard app-config directory.
